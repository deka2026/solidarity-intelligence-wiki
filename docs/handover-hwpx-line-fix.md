## 한 줄 요약

## 문제와 원인

- 증상: docx를 한글로 옮기면 각 ☑ 섹션 제목 위쪽과 그 다음 문단 첫 줄을 관통하는 회색 선이 나타남 (인쇄 미리보기에도 그대로 출력)
- 원인: 워드에서 제목 아래 장식용으로 넣은 얇은 남색 밑줄(`<w:pBdr><w:bottom .../>`)이 한글 변환 시 **사방 회색 테두리 박스(borderFill, #666666)**로 잘못 매핑되고, 한글이 그 박스의 위·아래 변을 어긋난 위치(다음 문단 첫 줄 관통)에 그림
- 확정 근거: hwpx 압축을 풀어 `Preview/PrvImage.png`(한글이 저장 시 직접 렌더링한 미리보기)를 열어보니 선이 정확히 ☑ 제목 문단 주변에만 있었고, 해당 제목 문단의 paraPr이 전변 SOLID borderFill을 참조하고 있었음

## 스킬

### 1. hwpx/docx는 둘 다 zip+XML — 직접 수술 가능
- docx: `word/document.xml`(본문)·`styles.xml`(스타일). hwpx: `Contents/header.xml`(charPr/paraPr/borderFill 정의)·`section0.xml`(본문)·`Preview/PrvImage.png`(한글이 렌더링한 미리보기 스냅숏)
- **PrvImage.png가 최고의 진단 도구**: 한글 없이도 "한글이 이 파일을 어떻게 그리는지"를 그대로 보여준다. Read 도구로 바로 열람
- hwpx 서식 추적 경로: 본문 `<hp:p paraPrIDRef="N">` → header의 `<hh:paraPr id="N">` → `<hh:border borderFillIDRef="M">` → `<hh:borderFill id="M">`의 4변 타입

### 2. Windows(파이썬 스텁 환경)에서 zip 재포장
- 이 PC의 `python`은 WindowsApps 스텁이라 실행 불가 → PowerShell .NET(`System.IO.Compression`) 또는 node로 대체
- docx는 일반 zip이면 됨(ZipArchive로 충분). **hwpx는 OCF 규칙**: `mimetype`이 첫 엔트리 + 무압축(Stored)이어야 안전
- 함정: .NET Framework `CreateEntry(name, NoCompression)`은 Stored가 아니라 Deflate로 기록됨(19→24바이트로 확인). 해결: node로 zip 로컬헤더·중앙디렉터리·CRC32를 직접 쓰는 미니 패커 작성(`pack-hwpx.js`, zlib.deflateRawSync + 수동 CRC 테이블)
- 한글 경로가 든 .ps1은 **UTF-8 BOM 필수** — BOM 없으면 PS 5.1이 ANSI로 읽어 경로가 깨짐("Illegal characters in path")

### 3. 잠긴(열려 있는) 파일 검사
- 한글에서 열린 hwpx는 ExtractToDirectory가 실패하지만 `Copy-Item`은 됨 → 사본을 떠서 검사
- 파일 잠김 자체가 "사용자가 지금 이 파일을 보고 있다"는 단서가 됨

### 4. hwpx 텍스트 추출·교정 파이프라인
- `<hp:p>` 단위로 내부 `<hp:t>`들을 이어붙이면 문단 단위 원문 복원 → 교정 검토에 사용
- 표 숫자는 반드시 검산(연차 합계·부문 합계·총계 교차 확인) — 이번 문서는 전부 일치했음

## 레슨

1. **"화면 표시"와 "파일 서식"을 먼저 구분하라** — 판별 기준은 인쇄 미리보기/PDF 출력. 이번에 맞춤법 도우미로 오진했다가 사용자의 "미리보기에도 나옴" 한 마디로 뒤집혔다. 처음부터 "미리보기에도 나오나요?"를 확인했으면 한 라운드를 아꼈다.
2. **XML에서 안 보이면 렌더링 산출물(PrvImage)을 봐라** — 서식 요소를 grep으로 다 뒤져도 원인이 안 나올 때, 한글이 직접 그린 미리보기 이미지가 "선이 정확히 어디에 있는지"를 알려줘 수색 범위를 문단 하나로 좁혀줬다.
3. **워드→한글 변환에서 문단 테두리(pBdr)는 위험 서식** — 제목 밑줄 장식은 한글에서 박스로 깨진다. 한글로 넘길 문서는 pBdr 대신 굵기·색으로 위계를 주는 게 안전하다.
4. **다른 도구가 만든 파일 규격은 원본과 비교 검증** — 새로 만든 hwpx의 mimetype이 Deflate로 저장된 걸, 한글이 직접 만든 hwpx(Stored)와 비교해서 잡아냈다.
5. **교정 제안은 위치+현재+수정안 3열로** — 사용자가 파일에서 바로 찾아 고칠 수 있는 형태가 실용적이다.
