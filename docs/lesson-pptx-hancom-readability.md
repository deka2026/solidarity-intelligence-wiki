# 레슨 — PPTX 산출물: 뷰어 호환·가독성·파일 잠금

**작성일:** 2026-07-16 | 관련: [[handover-youth-policy-pptx]], [[skill-pptxgenjs-hancom-compat]]

## 1. 산출물은 사용자의 실제 뷰어 기준으로 검증하라

pptxgenjs 산출물은 PowerPoint에서 정상이지만 한쇼는 "손상된 문서"로 판정했다. 이 사무실의 기본 뷰어는 한글·한쇼 계열이다. hwpx 때(한글)와 같은 교훈의 반복 — **"열리는 파일"의 기준은 만든 도구가 아니라 받는 사람의 프로그램**이다. PPTX를 만들면 한쇼 호환 후처리(노트 제거)를 기본으로 깔 것.

## 2. 사용자가 파일을 열어둔 채 재생성 요청하는 경우가 기본값이다

"다시 만들어줘"는 대부분 방금 만든 파일을 한쇼로 보면서 하는 말이다 → 덮어쓰기가 EBUSY로 실패한다. temp 생성 후 copy, 실패 시 대체 이름 저장 패턴을 처음부터 넣어야 하고, **어느 파일이 최신인지 매번 명시**해야 한다. 이번에 `_개선`, `버전2_개선` 파일이 생기며 한때 4개 파일 중 2개가 구버전인 혼란이 있었다 — 파일이 닫힌 뒤 정식 이름으로 재통합하는 마무리까지가 작업이다.

## 3. HTML 슬라이드의 1:1 이식은 여백과 잔글씨를 낳는다

HTML(1400×800px, 브라우저 스크롤 없는 화면용)의 비례를 그대로 PPT 캔버스에 옮기면 글자가 작고(11pt) 카드 안 공백이 크다. 사용자 피드백이 두 번 연속 "여백 줄여라, 글자 키워라, 대비 높여라"였다. **처음부터 PPT 기준으로 재설계**할 것: 본문 13.5pt 이상, 좌우 여백 0.4", 박스는 내용 직후에 붙이고 남는 높이는 박스 확대로 흡수.

## 4. 다크 테마 대비는 웹 기준보다 한 단계 더 올려라

웹에서 쓰는 slate-300(CBD5E1) 본문색이 프로젝터/PPT에서는 흐리다. 다크 배경 본문은 거의 흰색(F1F5F9), 강조 박스는 "짙은 배경 + 밝은 굵은 글자", 카드 테두리도 밝게. 라이트 테마 요청이 결국 왔다 — **처음부터 다크/라이트 두 벌을 물어보거나 팔레트만 갈아끼우는 구조로 스크립트를 짜면** 두 번째 요청이 5분 거리가 된다 (이번에 팔레트 상수만 교체해서 라이트 2종을 바로 냈다).

## 5. 재생성 스크립트는 scratchpad에 두지 말고 보존하라

생성 스크립트가 세션 임시폴더에만 있으면 다음 수정 요청 때 처음부터 다시 짜야 한다. 슬라이드 텍스트 전문이 스크립트 안에 있으므로 스크립트가 곧 원본 — `클로드로컬\pptx-scripts\`에 복사해 두었다. 마이크로그리드 해설집 때의 "재생성 수단 남기기" 교훈과 동일.

## 6. 원인 진단은 XML을 직접 까서 확인하라

"한쇼가 까다롭다"로 끝내지 않고 presentation.xml을 열어 notesMasterIdLst 위치가 스키마 위반임을 확인했기에, 재발 방지(후처리 상시화)까지 갈 수 있었다. python 검증기가 없어도 PowerShell ZipFile + XmlDocument로 최소한의 구조 검증은 가능하다.
